# 코드를 자동으로 배포하기 — CI/CD

## CI/CD가 뭔가요?

**CI/CD = "자동 배포 시스템"**

구글독스를 생각해보세요. 파일을 수정하면 저장 버튼 안 눌러도 자동으로 반영됩니다.

**CI/CD도 비슷합니다:**
```
당신이 코드 수정
→ GitHub에 업로드
→ 자동으로 검사 (버그 없나?)
→ 자동으로 배포 (서버에 반영)
→ 끝!
```

**비유:**
- **수동 배포**: 음식을 매번 손으로 만들어서 테이블에 갖다주기 (느림, 실수 많음)
- **CI/CD**: 자동 조리 로봇이 음식을 만들고, 배달 로봇이 갖다주기 (빠름, 실수 적음)

---

## 왜 필요한가요?

### 없을 때의 문제

```
당신이 해야 할 일:
1. 코드 수정
2. 로컬에서 테스트
3. 서버 접속
4. 파일 삭제
5. 새 파일 업로드
6. 서버 재시작
7. 다시 테스트
8. 문제 생기면? 이전 버전 찾아서 복구...

→ 30분 걸림, 실수 많음
```

### CI/CD 있을 때의 과정

```
당신이 해야 할 일:
1. 코드 수정
2. GitHub에 업로드 (push)

→ 자동으로 나머지 처리
→ 5분 걸림, 실수 거의 없음
```

**효과:**
- 시간 절약: 85% 단축
- 실수 감소: 자동 검사로 버그 조기 발견
- 마음의 평화: 배포가 자동이니 걱정 덜어짐

---

## 우리 회사 방식

### 3단계 자동화

```
1️⃣  코드 수정 및 업로드
   당신: "코드 완성했어, push"

2️⃣  자동 검사 & 테스트
   자동: "이 코드 버그 없나? 테스트 돌려봐"

3️⃣  자동 배포
   자동: "좋아, 서버에 올릴게"

✅ 끝! 사용자가 사용 가능
```

### 어디서 확인?

```
GitHub → Actions 탭 → 초록색(성공) 또는 빨간색(실패)
```

## AI한테 맡기기

### 새 프로젝트에 CI/CD 설정하기

프로젝트 처음 시작할 때:

```
이 프로젝트에 자동 배포를 설정해줄 수 있을까?
```

또는 명령어로:

```bash
/cicd-setup
```

**AI가 할 일:**
1. GitHub 설정 (자동 실행 규칙 추가)
2. 서버 설정 (배포 파일 업로드)
3. 테스트 (제대로 작동하나?)
4. 문서 작성 (배포 방법 설명)

**결과:**
- `.github/workflows/` 폴더 생성 (자동화 규칙)
- `deploy.sh` 생성 (배포 스크립트)
- README에 배포 가이드 추가

## 배포 확인하기

### GitHub에서 확인

1. GitHub 사이트 방문
2. 프로젝트 선택
3. **Actions** 탭 클릭

```
✅ 초록색 체크마크
   → 배포 성공!

❌ 빨간색 X 마크
   → 배포 실패 (뭐가 문제일까?)
```

### 상세 로그 보기

초록색/빨간색 항목 클릭 → 상세 로그 확인

```
예시:
─────────────────
✅ 테스트 통과
✅ 빌드 성공
✅ 배포 완료
```

또는:

```
❌ 테스트 실패
   - TypeError: undefined is not a function (src/api.js:45)
```

### AI한테 물어보기

**배포 실패했을 때:**

```
GitHub Actions에서 배포가 실패했어. 이 에러를 확인해줄 수 있을까?
```

(에러 메시지 붙여주기)

AI가:
1. 에러 원인 분석
2. 어디서 문제인지 가리키기
3. 고쳐진 코드 제시
4. 다시 배포

## 문제 생기면?

### 배포 자동 실패 처리

배포가 실패하면:

```
💡 안내:
 - 이전 버전으로 자동 복구됨
 - 사용자는 영향 없음
 - 당신은 에러 메시지 확인 후 수정
```

**타임라인:**
```
1:00 PM   배포 시작
1:05 PM   테스트 실패 ❌
1:06 PM   자동으로 이전 버전 복구 ✅
          (사용자는 전혀 모름)
1:30 PM   당신이 버그 수정
1:35 PM   다시 배포 ✅
```

---

## ⚠️ 주의사항

### 1. 배포 설정은 신중하게

```
❌ 위험:
"아무나 배포할 수 있게 해줄래"

✅ 안전:
"팀장만 배포 권한 가지게 해줄래"
```

회사 규정에 따라 개발팀과 상의하세요.

---

### 2. 환경 변수 관리

```
❌ 안 됨:
API_KEY="sk_live_xxx"를 코드에 직접 쓰기

✅ 좋음:
API_KEY를 GitHub Secrets에 저장
코드에서는 ${API_KEY} 참조
```

**AI에게 말하기:**
```
API 키를 안전하게 설정해줄 수 있을까?
```

---

### 3. 배포 전 테스트 꼭 하기

```
❌ 위험:
코드만 수정하고 테스트 안 함
→ 배포 실패 → 사용자 피해

✅ 안전:
로컬에서 테스트 → CI/CD 자동 테스트
→ 배포
```

## 배포 로그 이해하기

### 성공한 배포 로그

```
[main 1a2b3c4] feat: add login page
 1 file changed, 50 insertions(+)

✅ GitHub Actions
   └─ tests      ... PASSED
   └─ build      ... PASSED
   └─ deploy     ... PASSED

✨ 배포 완료 (prod 서버)
```

### 실패한 배포 로그

```
❌ GitHub Actions
   └─ tests      ... FAILED
      └─ Error: Expected true, got false
         File: tests/login.test.js:25

🔧 조치:
   - 테스트 코드 수정 필요
   - 다시 push
```

```

→ 배포 함
```

```
AI한테 물어보기:

```

## 배포 실패했을 때 AI한테 물어보기

```
GitHub Actions에서 배포가 실패했어.
에러: "npm ERR! Cannot find module 'express'"

이게 뭐고 어떻게 고쳐?
```

AI가:
1. 문제 분석: "package.json에 express가 없음"
2. 원인: "dependencies에 안 적혔음"
3. 해결: "npm install express 또는 package.json 수정"
4. 코드 제시
